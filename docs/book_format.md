# Book of Infinite Tales — Format Reference

This document describes the complete format for a Book of Tales: the JSON files that define a book's passages and game components. It is written for an agent generating or validating book content.

---

## Overview

A Book of Tales is a supplement for the board game *Tales of the Arthurian Knights*. It is a set of numbered passages (like a Choose Your Own Adventure) where encounters are looked up by passage number, players read prose, make narrative choices, attempt skill checks, and receive rewards. The book contains no state — it is a pure navigation and narrative structure.

A book lives in a GitHub repository and is loaded by the **Book of Infinite Tales** reader app at runtime.

---

## Repository structure

A minimal book needs one file:

```
book.json
```

A book with the encounter picker needs two:

```
book.json
components.json      (can be named anything; book.json references it)
```

Multiple books can share one components file:

```
my-repo/
  components.json
  book-one/book.json
  book-two/book.json
```

Each `book.json` references the shared file with a relative path: `"components": "../components.json"`.

---

## `book.json` — the book manifest

```json
{
  "schema": "book-of-infinite-tales/v1",
  "title": "Your Book Title",
  "author": "Your Name",
  "version": "1.0.0",
  "description": "One sentence about this book.",
  "components": "components.json",
  "aiGenerated": true,
  "entries": [ ... ]
}
```

| Field | Required | Description |
|---|---|---|
| `schema` | Yes | Must be exactly `"book-of-infinite-tales/v1"` |
| `title` | Yes | Display name of the book |
| `author` | No | Author's name |
| `version` | No | Semver string, e.g. `"1.0.0"` |
| `description` | No | One-sentence summary |
| `components` | No | Filename of the components file (relative to `book.json`). Omit if you don't use the encounter picker. |
| `aiGenerated` | No | Set to `true` if the book's content was wholly or partly generated by an AI tool. The reader displays a visible notice so players can make an informed choice about whether to use the book. |
| `entries` | Yes | The passages. Either an inline array of entry objects, an inline `id → entry` object map, or a filename string pointing to a separate JSON file. |

### Entries as a separate file

For large books, `"entries": "passages.json"` is valid. The referenced file contains either an array or an `id → entry` map.

---

## Passages (entries)

Every passage is an **entry**. Each entry has an `id` and `body`. The other fields determine what type of passage it is and what the player can do.

### Shared fields (all entries)

| Field | Type | Required | Description |
|---|---|---|---|
| `id` | string | Yes | Unique identifier. Numeric strings (`"1234"`) sort naturally in the reader. |
| `body` | string | Yes | Prose read aloud. Use `\n\n` to separate paragraphs. |
| `romantic` | boolean | No | Marks the entire entry as romantic content. |
| `retinue` | string | No | Whether the knight's retinue is present. One of `"beside"`, `"nearby"`, or `"absent"`. |
| `responses` | array | No | Narrative choices (see Response passages). |
| `resolutions` | array | No | Skill check options (see Resolution passages). |
| `rewards` | object | No | Rewards applied at this passage (see Rewards). |
| `goto` | string | No | Entry id to jump to after reading this passage. |

### Passage types

The presence of these fields determines the passage type. Types can be mixed for unusual encounter flows.

| Type | Key fields | Description |
|---|---|---|
| **Response passage** | `responses` | Presents narrative choices. Each choice points to another entry. Usually the opening passage of an encounter. |
| **Resolution passage** | `resolutions` | Presents skill check options. Each option has inline success and failure outcomes. |
| **Result passage** | `rewards` and/or plain `body` with no `responses`/`resolutions` | Terminal passage. Read the prose and apply rewards. |
| **Unusual encounter** | Any mix | Some encounters combine elements, e.g. `resolutions` on the first passage, or `rewards` plus `goto` for a continuation. |

**Age start, Epilogue, and Quest passages are always Result passages.** They contain narrative prose and rewards, but never `responses`, `resolutions`, or `goto`. Age start passages mark the beginning of an age; the Epilogue marks the end of the game; Quest passages mark the completion of a quest.

**Quest and Status encounter passages defer to the physical book.** Quest card text and status encounter text are printed in the official Book of Tales and cannot be reproduced here. Every quest passage and every status encounter passage must use exactly this body text:

```
"Refer to physical Book of Tales for this passage."
```

No rewards, no additional prose. The body is the entire entry.

**Result passages as encounter exits are extremely rare.** Outside of age starts, epilogues, and quests, a result passage appears only when a player has chosen to disengage entirely from an encounter — the rare 3rd option on a response passage. Such a passage must include a meaningful negative reward (a loss of destiny, a negative renown delta, or an unwanted status) to reflect the cost of not engaging with the story. Do not use result passages as a convenient shortcut for encounters that are hard to resolve through skill checks.

## Response passages

A response passage offers narrative choices. Each choice (a `ResponseOption`) points to another entry via `goto`.

```json
{
  "id": "1500",
  "body": "You come upon an old well...",
  "responses": [
    {
      "label": "You may call down into the well.",
      "goto": "1501"
    },
    {
      "label": "* You may greet the voice tenderly.",
      "romantic": true,
      "goto": "1503"
    }
  ]
}
```

### ResponseOption fields

| Field | Type | Required | Description |
|---|---|---|---|
| `label` | string | Yes | The choice text shown to the player. |
| `goto` | string | Yes | Entry id to navigate to when this choice is made. |
| `romantic` | boolean | No | Marks this option as romantic content. Convention: prefix label with `*`. |

Responses never contain skill checks — those happen on the resolution passage the response points to.

### Response passage length

The `body` of a response passage should be **2 to 5 paragraphs**. Use `\n\n` to separate paragraphs. This is the scene-setting prose that draws the player in before they make their choice — it should be rich enough to ground the encounter but not so long it buries the decision.

### Number of response options

Most response passages should offer exactly **2 options**. Both should lead to resolution passages that fully engage with the story.

A **3rd option** is permitted only as an exit — a choice to disengage from the encounter entirely, pointing to a result passage rather than a resolution passage. This exit option should be **extremely rare**. It represents a deliberate narrative choice to walk away, and it must carry an appropriate negative consequence (see Result passages below). Do not include an exit option by default; only add one when it is narratively meaningful.

---

## Resolution passages

A resolution passage is where a skill check occurs. It has one or more `ResolutionOption` entries. The player picks one option, commits, then rolls (or compares renown) against the target.

```json
{
  "id": "1501",
  "body": "The voice calls up from below: 'Who asks?'",
  "resolutions": [
    {
      "label": "answer with your true name",
      "using": ["Piety", "Honor"],
      "target": 3,
      "success": {
        "body": "You give your name without hesitation...",
        "rewards": { "destiny": 2, "renown": [{ "type": "Divinity", "delta": 1 }] }
      },
      "failure": {
        "body": "You cannot find the words...",
        "rewards": { "destiny": 1, "renown": [{ "type": "Divinity", "delta": -1 }] }
      }
    },
    {
      "label": "puzzle out the meaning before speaking",
      "using": ["Wisdom", "Magic", "Spiritual"],
      "target": { "base": 2, "addLocationNumber": true },
      "success": {
        "body": "You answer with the old forms...",
        "rewards": { "destiny": "location_number", "treasures": "Ring of the Well" }
      },
      "failure": {
        "body": "You take too long...",
        "rewards": { "destiny": 1, "statuses": [{ "action": "gain", "name": "Obsessed" }] }
      }
    }
  ]
}
```

### ResolutionOption fields

| Field | Type | Required | Description |
|---|---|---|---|
| `label` | string | No | Narrative description of what this skill choice represents. |
| `using` | array of strings | Yes | Exactly one entry: either a single skill name or a single skill category. See below. |
| `target` | number or object | Yes | The difficulty (see Resolution targets). |
| `romantic` | boolean | No | Marks this option as romantic content. |
| `success` | ResolutionOutcome | Yes | Outcome when the player meets or exceeds the target. |
| `failure` | ResolutionOutcome | Yes | Outcome when the player fails to meet the target. |

### Resolution skill choices

The `using` array must contain **exactly one entry**, which is either:
- A **single specific skill** name (e.g. `["Piety"]`) — use this when the narrative calls for a precise approach.
- A **single skill category** (e.g. `["Martial"]`) — use this when any skill within the category would plausibly fit the action.

Do not list multiple individual skills in a single `using` array. If a resolution option could logically use more than one skill, use the skill category instead.

### Resolution passage length

- The `body` of the resolution passage (before the skill check) should be **1 to 3 paragraphs**.
- The `body` of each outcome (`success` and `failure`) should also be **1 to 3 paragraphs**.

### ResolutionOutcome fields

| Field | Type | Required | Description |
|---|---|---|---|
| `body` | string | Yes | Prose read when this outcome occurs. |
| `rewards` | Reward object | No | Rewards applied for this outcome. |
| `goto` | string | No | Entry id to continue to after this outcome. Rare — most outcomes are terminal. |

---

## Check options (`using` array values)

The `using` array accepts three kinds of value:

### Skill names (exact strings, case-sensitive)

| Martial | Spiritual | Courtly | Wilderness |
|---|---|---|---|
| `Warfare` | `Piety` | `Diplomacy` | `Nature Lore` |
| `Sword & Shield` | `Wisdom` | `Cunning` | `Endure Hardship` |
| `Mounted` | `Magic` | `Honor` | `Hunting` |

### Skill categories

Use a category to allow any skill within it:

- `"Martial"` — Warfare, Sword & Shield, Mounted
- `"Spiritual"` — Piety, Wisdom, Magic
- `"Courtly"` — Diplomacy, Cunning, Honor
- `"Wilderness"` — Nature Lore, Endure Hardship, Hunting

### Renown types

Use a renown type for a renown check (no die roll — the player compares their accumulated ranks to the target). Valid values are the `name` strings declared in `components.renown`, plus the special keyword `"Any"` (player chooses which track to use). For the standard game: `"Divinity"`, `"Romance"`, `"Villainy"`, `"Any"`.

---

## Resolution targets

| Format | Meaning |
|---|---|
| A plain number, e.g. `3` | Fixed difficulty: roll 1d6 + Skill Rank ≥ this value |
| `{ "base": N, "addLocationNumber": true }` | Variable: difficulty = `N` + the Location # of the knight's current map space (a value 1–6 printed on the board space) |

The variable form is commonly used for milieu passages where dangerous terrain scales with location.

---

## Rewards

A `Reward` object appears on result passages and on resolution outcomes. All fields are optional.

```json
"rewards": {
  "destiny": 2,
  "renown": [
    { "type": "Divinity", "delta": 1 },
    { "type": "Villainy", "delta": -1 }
  ],
  "skills": [
    { "name": "Piety" },
    { "category": "Spiritual", "count": 1 }
  ],
  "treasures": 1,
  "statuses": [
    { "action": "gain", "name": "Accompanied" },
    { "action": "lose", "name": "Obsessed" }
  ],
  "storyToken": 3,
  "movement": 2
}
```

| Field | Type | Description |
|---|---|---|
| `destiny` | number or `"location_number"` | Destiny Points gained. `"location_number"` = gain points equal to the knight's current Location # (1–6). |
| `renown` | array of `{type, delta}` | Renown changes. `delta` is positive to gain ranks, negative to lose. `type` is a renown name from `components.renown`, or `"Any"` for player's choice of track. |
| `skills` | array | Skill rewards. `{"name": "Piety"}` = gain that specific skill. `{"category": "Spiritual", "count": 1}` = gain any one Spiritual skill of the player's choice. `count` defaults to 1. |
| `treasures` | number or string | Number = draw that many random Treasure cards. String = search for the named Treasure card specifically. |
| `statuses` | array of `{action, name}` | `"gain"` = acquire the named Status card. `"lose"` = discard the named Status card early. |
| `storyToken` | number | The Story Token number gained. |
| `movement` | number or `"free"` | Bonus map movement after the encounter. `"free"` = move anywhere. |

### Design principle: failures still reward

In the Arthurian tradition, every outcome — success and failure — should grant something positive alongside any negative consequence. Two rules apply to all failure outcomes:

1. **Failed skill checks award a skill gain.** When a resolution option uses a specific skill or skill category and the player fails, the failure reward must include a `skills` gain for that skill or category — the knight learned something from the attempt. Example: a failed `["Piety"]` check gives `"skills": [{ "name": "Piety" }]`; a failed `["Martial"]` check gives `"skills": [{ "category": "Martial" }]`.

2. **Negative consequences come from other reward types.** Failures are distinguished from successes through renown loss, unwanted statuses, or withheld rewards.

## Retinue presence

```json
"retinue": "beside"
```

Optional scene-setting note about whether the knight's retainers are present.

| Value | Meaning |
|---|---|
| `"beside"` | Retainers are right there with the knight |
| `"nearby"` | Retainers are close but not at the knight's side |
| `"absent"` | Knight is scouting ahead; retainers are elsewhere |

---

## Romantic content

Mark optional romantic content at three levels of granularity:

- On the **entry**: `"romantic": true` — the passage's body text is romantic. The reader marks the entire passage, including the prose, as romantic content.
- On a **response option**: `"romantic": true` — just this choice and the passage it leads to are romantic.
- On a **resolution option**: `"romantic": true` — just this skill option is romantic.

Convention: prefix the `label` of a romantic response or resolution option with `*` (e.g. `"* You may greet the voice as a beloved."`). The reader renders romantic content in a distinctive colour; players may skip it.

---

## `components.json` — game components

The components file describes the physical game components. It enables the **encounter picker** in the reader, which lets players select their encounter type and navigates directly to the correct passage number. A book that omits `"components"` from its manifest still works — players type passage numbers manually.

All top-level fields in the components file are optional. Include only what your book uses.

```json
{
  "ages": [...],
  "terrains": [...],
  "features": [...],
  "characters": [...],
  "locations": [...],
  "milieus": [...],
  "quests": [...],
  "renown": [...],
  "skills": [...],
  "storyTokens": [...],
  "statuses": [...],
  "epiloguePassage": "9999"
}
```

---

### Ages

```json
"ages": [
  { "id": "1", "name": "Golden Age of Camelot",   "startPassage": "1000", "milieuBase": 2200 },
  { "id": "2", "name": "Quest of the Holy Grail",  "startPassage": "2000", "milieuBase": 2300 },
  { "id": "3", "name": "Final Wars of Britain",    "startPassage": "3000", "milieuBase": 2400 }
]
```

| Field | Required | Description |
|---|---|---|
| `id` | Yes | Short identifier, e.g. `"1"`, `"2"`, `"3"`. Used as a key in `visitPassages`. |
| `name` | Yes | Full display name. |
| `startPassage` | No | Entry id read aloud when this age begins. |
| `milieuBase` | No | Base number added to a milieu's terrain offset to form the milieu passage id. Required if the book uses Milieu encounters. |

---

### Terrains

```json
"terrains": [
  { "id": "city",     "name": "City" },
  { "id": "forest",   "name": "Forest" },
  { "id": "mountain", "name": "Mountain" },
  { "id": "plains",   "name": "Plains" },
  { "id": "swamp",    "name": "Swamp" },
  { "id": "sea",      "name": "Sea" }
]
```

Used as keys in `milieu.terrainOffsets`. Declare only the terrains your milieus support. The standard game has six: City, Forest, Mountain, Plains, Swamp, Sea.

---

### Characters and Features

Characters and Features combine to produce a passage number:

```
passage = character.base + feature.offset
```

```json
"characters": [
  { "id": "king",   "name": "King",   "base": 1000 },
  { "id": "queen",  "name": "Queen",  "base": 1100 },
  { "id": "knight", "name": "Knight", "base": 1200 }
],
"features": [
  { "id": "wandering", "name": "Wandering", "offset": 1  },
  { "id": "sorrowful", "name": "Sorrowful", "offset": 2  },
  { "id": "amorous",   "name": "Amorous",   "offset": 5  },
  { "id": "mad",       "name": "Mad",       "offset": 7  }
]
```

**Character** fields: `id`, `name`, `base` (a multiple of 100, e.g. 1000, 1100, 1200).

**Feature** fields: `id`, `name`, `offset` (a 2-digit number, e.g. 1, 5, 12, 20).

Example: Knight (`base: 1200`) + Amorous (`offset: 5`) → passage `1205`. That entry must exist in the book.

Each character "owns" a block of 100 passage numbers. Features share the same offsets across all characters. The standard game has 12 characters and 11 features, producing 132 required passages.

---

### Locations

```json
"locations": [
  {
    "id": "avalon",
    "name": "Avalon",
    "passage": "2601",
    "visitPassages": {
      "1": "2501",
      "2": "2513",
      "3": "2525"
    }
  }
]
```

| Field | Required | Description |
|---|---|---|
| `id` | Yes | Unique identifier. |
| `name` | Yes | Display name. |
| `passage` | Yes | Entry id read when the Location card is first drawn from the deck. |
| `visitPassages` | No | Map of `age.id → entry id` for the Place of Power encounter. The entry read when the knight visits this location as a Special Encounter during the given age. |

A location produces **one regular passage** plus **one visit passage per age** that has an entry in `visitPassages`. These passage numbers must all be distinct — the `passage` (card draw) and the `visitPassages` entries are separate encounters and must not share a passage id.

---

### Milieus

```json
"milieus": [
  {
    "id": "strange-beast",
    "name": "Strange Beast",
    "terrainOffsets": {
      "city":     5,
      "forest":   17,
      "mountain": 29,
      "plains":   41,
      "swamp":    53,
      "sea":      65
    }
  }
]
```

The milieu passage id is computed as:

```
passage = age.milieuBase + milieu.terrainOffsets[terrain.id]
```

Example: Age 1 (`milieuBase: 2200`) + Strange Beast + Mountain (`offset: 29`) → passage `2229`.

| Field | Required | Description |
|---|---|---|
| `id` | Yes | Unique identifier. |
| `name` | Yes | Display name. |
| `terrainOffsets` | Yes | Map of `terrain.id → 2-digit offset`. Terrains not listed are unavailable for this milieu. |

Each milieu produces one passage per (age, terrain) combination where the terrain has an offset. With 3 ages, 9 milieus, and 6 terrains, the standard game produces 162 milieu passages.

---

### Quests

```json
"quests": [
  { "id": "dragon-slayer", "name": "Dragon Slayer", "passage": "1009" }
]
```

| Field | Required | Description |
|---|---|---|
| `id` | Yes | Unique identifier. |
| `name` | Yes | Display name shown in the picker. |
| `passage` | Yes | Entry id navigated to when this quest is selected. |

---

### Treasures

```json
"treasures": [
  { "id": "holy-lance",    "name": "Holy Lance"    },
  { "id": "clarent",       "name": "Clarent"       },
  { "id": "faerie-ring",   "name": "Faerie Ring"   }
]
```

Treasure cards are physical components of the game. Declaring them in the components file enables validation: if a reward uses a named treasure string (e.g. `"treasures": "Holy Lance"`), the loader will reject any name not in this list.

| Field | Required | Description |
|---|---|---|
| `id` | Yes | Unique identifier. |
| `name` | Yes | Display name used in `reward.treasures` string values. Must match exactly (case-sensitive). |

When `reward.treasures` is a **number**, the player draws that many random Treasure cards — no name lookup is needed. When it is a **string**, it must match a declared treasure name and the player searches for that specific card.

---

### Renown

```json
"renown": [
  { "id": "divinity", "name": "Divinity" },
  { "id": "romance",  "name": "Romance"  },
  { "id": "villainy", "name": "Villainy" }
]
```

Renown tracks are physical components of the game — each player has a track on their player board for each declared renown type. Declaring renown in the components file makes the valid track names explicit and enables validation of `type` values in rewards and `using` values in resolutions.

| Field | Required | Description |
|---|---|---|
| `id` | Yes | Unique identifier. |
| `name` | Yes | Display name used in `renown[].type` reward fields and `using` arrays. Must match exactly (case-sensitive). |

The special value `"Any"` is not declared as a renown entry — it is a built-in keyword meaning the player chooses which track to apply. Use it in `using` arrays and reward `type` fields when the book does not specify a track.

---

### Skills

```json
"skills": [
  { "id": "warfare",   "name": "Warfare",        "category": "Martial"    },
  { "id": "piety",     "name": "Piety",           "category": "Spiritual"  },
  { "id": "diplomacy", "name": "Diplomacy",       "category": "Courtly"    },
  { "id": "hunting",   "name": "Hunting",         "category": "Wilderness" }
]
```

Declaring skills in the components file enables validation: the loader will reject any `using` value in a resolution that is not a declared skill name, skill category, or renown type. `category` must be one of `"Martial"`, `"Spiritual"`, `"Courtly"`, `"Wilderness"`.

---

### Story tokens

```json
"storyTokens": [
  { "number": 1 },
  { "number": 4, "note": "Passage 2134" },
  { "number": 5, "note": "Age 3 → Token 6" }
]
```

Story tokens are numbered 1–N. `note` records any printed effect on the physical token, for reference. When `storyTokens` is declared, the loader validates that `reward.storyToken` values refer to declared token numbers.

---

### Statuses

```json
"statuses": [
  { "id": "accompanied", "name": "Accompanied" },
  {
    "id": "imprisoned",
    "name": "Imprisoned",
    "encounters": [
      { "label": "Attempt to escape", "passage": "2537" },
      { "label": "Attempt to rescue", "passage": "2544" }
    ]
  }
]
```

Status cards name conditions a knight can hold during play. Most statuses have no `encounters`. When `encounters` is present, those passages are accessible via the **Status** button in the Special Encounters section of the picker — the player selects their status and picks the relevant encounter.

| Field | Required | Description |
|---|---|---|
| `id` | Yes | Unique identifier. |
| `name` | Yes | Display name. |
| `encounters` | No | Array of `{label, passage}` — passages associated with holding this status. |

---

### Epilogue

```json
"epiloguePassage": "9999"
```

A single entry id read at the very end of the game. The reader shows an Epilogue button alongside the age buttons.

## Passage numbering conventions

The standard game uses these conventions, which your book should follow to feel consistent:

| Range | Usage |
|---|---|
| `1000`, `2000`, `3000` | Age start passages |
| `1xxx`–`1xxx` | Age 1 character encounters (base 1000–2099) |
| `2200`–`2272` | Age 1 milieu encounters (milieuBase 2200) |
| `2300`–`2372` | Age 2 milieu encounters (milieuBase 2300) |
| `2400`–`2472` | Age 3 milieu encounters (milieuBase 2400) |
| `2501`–`2536` | Place of Power visit passages (12 locations × 3 ages) |
| `2601`–`2612` | Location card-draw passages (12 locations) |
| `1009`, `1011`, ... | Quest passages (scattered in the character number ranges) |
| `9999` | Epilogue |

**Key rule:** every passage id computed by the components rules must exist as an entry in the book, and every passage id must be globally unique across all rules.

### Passage numbering within an encounter

The response passage and all its resolution and result passages should be **scattered throughout the same thousand block**, not placed consecutively. For example, an encounter starting at `1347` might have resolution passages at `1372`, `1389`, and `1412` (still within the `1xxx` block for Age 1).

This is intentional: if related passages were consecutive, a player who accidentally glanced ahead or miscounted would read a passage that spoils an outcome. Scattering passages prevents this. Do not use consecutive numbering for the passages that make up a single encounter.

## Complete passage enumeration

To generate every required passage id from a components file, apply these rules in order:

1. **Age starts**: one per age with `startPassage` → `age.startPassage`
2. **Character encounters**: one per (character, feature) pair → `character.base + feature.offset`
3. **Location card draws**: one per location → `location.passage`
4. **Place of Power visits**: one per (location, age) pair where `visitPassages[age.id]` exists → `location.visitPassages[age.id]`
5. **Milieu encounters**: one per (age, milieu, terrain) triple where `terrainOffsets[terrain.id]` exists → `age.milieuBase + milieu.terrainOffsets[terrain.id]`
6. **Quests**: one per quest → `quest.passage`
7. **Status encounters**: one per (status, encounter) pair → `encounter.passage`
8. **Epilogue**: one if `epiloguePassage` is set → `epiloguePassage`

All generated passage ids must be unique. No two rules should produce the same number.

Use the script `scripts/generate-passage-list.py` to enumerate all required passages from a components file:

```
python scripts/generate-passage-list.py components.json
python scripts/generate-passage-list.py components.json --json
```

Output is one line per passage: `<id>\t<description>`. With `--json`, a JSON array of `{passage, description}` objects.
